# 目标不对齐检测实验报告
## Goal Misalignment in Multi-Agent Small Language Model Systems

> **研究问题**：当父代理（parent agent）将任务主动分解并委派给子代理（subagent）时，父子代理之间是否存在目标不对齐现象？这种现象在不同规模的模型（0.5B / 1.5B / 3B）上如何表现？
>
> **核心发现**：目标不对齐在所有模型上均存在，且随着模型规模增大而**加剧**（而非减弱）。委派任务分解的收益被目标不对齐损失部分或完全抵消。
>
> **日期**：2026-04-25 | **仓库**：RidenShogunei/tiny-agents | **commit**：d61f541

---

## 1. 研究背景与动机

### 1.1 多智能体系统的潜在优势与风险

基于小型 Qwen 模型（0.5B–3B）的多智能体框架具有内存效率和分布式推理的潜力。然而，多智能体协作存在一个根本性风险：**目标不对齐（Goal Misalignment）**——父代理期望子代理提供的知识/能力，与子代理实际能够提供的输出之间存在偏差，导致多智能体系统的整体表现低于单一代理。

这一现象在大型模型中已有研究（AgentAlign, MINT-Bench），但在小型模型上尚未被系统验证。

### 1.2 学术基准参考

本实验设计参考了两个重要的学术基准：

**MINT-Bench**（Multi-agent Interaction Benchmark）：
- 核心洞察：多步骤协作任务中，代理之间必须真实交互才能完成
- 关键发现：信息损失（information loss）、语义漂移（semantic drift）、优先级冲突（priority conflict）是三大不对齐来源

**AgentAlign**：
- 核心洞察：个体对齐（individual alignment）不等于系统对齐（system alignment）
- 关键发现：代理间目标不一致是系统性现象，不能通过提升单代理能力自动解决

### 1.3 研究问题

本实验聚焦于以下核心问题：

1. **存在性**：在 0.5B–3B 的小模型上，父子代理之间是否存在目标不对齐？
2. **量化**：目标不对齐导致的准确率损失有多大？
3. **分解**：不对齐损失在总损失中占比多少？与额外步骤成本如何区分？
4. **规模性**：目标不对齐是否随模型规模增大而改善？

## 2. 实验设计演进

### 2.1 版本迭代历程

实验设计经历了四个主要版本的迭代：

| 版本 | 核心设计 | 主要发现 |
|------|---------|---------|
| **v1** | 60 案例，6 种错位类型，三模式（直答/预言机委派/真实委派） | 发现额外步骤成本占主导（79%），目标不对齐仅占 21% |
| **v2** | 子进程隔离 GPU 执行，排除干扰 | 基线准确率过低，任务设计存在问题 |
| **v3** | 合成任务，更严格的 prompt 设计 | 在 3B 上发现更严重的 misalignment（+35%） |
| **v4** | 基于 MINT-Bench/AgentAlign 重新设计，34 案例，5 种任务类型 | **全模型确认目标不对齐存在，3B 最严重** |

### 2.2 版本间关键差异

```
v1 → v4 的核心改进：
├── v1 问题：任务过于简单（单步知识问答），额外步骤成本掩盖了 misalignment
├── v4 改进：多步骤推理、外部知识、格式规范等结构化任务
│
├── v1 问题：直答基线本身过低（~52%），使 C > B 的增量不显著
├── v4 改进：任务设计确保直答准确率在 30-40% 区间，便于观察委派效果
│
└── v1 问题：v1 中 C-B = 5pp，太小，难以得出强结论
    v4 改进：重新设计任务使 subagent 输出更结构化，放大 misalignment 信号
```

## 3. Benchmark v4 详细设计

### 3.1 三模式实验框架

Benchmark v4 采用三模式测量设计，能够将总损失分解为两个独立来源：

```
┌─────────────────────────────────────────────────────────────────┐
│  模式 A（直答基线）                                               │
│  问题 → 模型直答 → 准确率 A%                                      │
│                                                                  │
│  用途：测量无委派时的基线准确率                                     │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  模式 C（预言机委派）                                              │
│  问题 → 父代理向子代理提问 → 我们注入 oracle 答案 → 父代理综合 → C% │
│                                                                  │
│  oracle 答案 = 任务设计中预设的正确子代理输出                      │
│  用途：测量委派步骤本身的成本（排除子代理质量因素）                 │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  模式 B（真实委派）                                                │
│  问题 → 父代理向子代理提问 → 真实子代理回答 → 父代理综合 → B%      │
│                                                                  │
│  用途：测量真实多智能体协作时的最终准确率                          │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 损失分解公式

```
额外步骤损失（Extra-step Loss）= A% - C%
    → 即便子代理输出完美，额外增加一个委派步骤带来的准确率下降
    → 反映的是"小模型无法处理多步推理"的固有缺陷

目标不对齐损失（Misalignment Loss）= C% - B%
    → 预言机优于真实子代理的部分
    → 纯粹的目标不对齐：父代理无法正确使用子代理输出

总委派损失（Total Loss）= A% - B% = 额外步骤损失 + 目标不对齐损失
```

### 3.3 五种任务类型

Benchmark v4 设计了 5 种共 34 个任务，目标是**放大 misalignment 信号**：

```
Type 1: Multi-step Math（多步数学）— 10 题
    父代理将计算任务分解为子步骤
    示例：列车先以 60km/h 行驶 2 小时，再以 90km/h 行驶 1.5 小时，总距离？
    子代理计算第一段距离（120 km），父代理需要加上第二段

Type 2: Two-Part Questions（双部分问题）— 8 题
    父代理需要结合子代理答案与自己的知识
    示例：埃菲尔铁塔 330m，自由女神像从地面到火炬 93m，哪个更高？高多少？
    子代理给出自由女神像高度（93m），父代理需要用自己的知识（330m）做比较

Type 3: Constraint Satisfaction（约束满足）— 6 题
    子代理提供候选集，父代理施加额外过滤条件
    示例：在 1-20 中找能被 3 整除但不能被 2 整除且小于 15 的最大数
    子代理列出所有能被 3 整除的数，父代理需要应用全部约束条件

Type 4: Comparative Analysis（比较分析）— 5 题
    子代理计算一个选项，父代理需要比较两个选项
    示例：3/5 和 4/7 哪个更大？
    子代理将 3/5 转换为小数（0.600），父代理需要自己计算 4/7 并比较

Type 5: Sequential Logic（序列逻辑）— 5 题
    子代理输出是父代理下一步推理的前提
    示例：如果所有 A 都是 B，没有 B 是 C，能否推断 A 和 C 的关系？
    子代理回答 B 和 C 的关系，父代理需要结合"所有 A 是 B"做出推论
```

### 3.4 评估方法

采用多策略评估：
- **精确匹配**：标准化数字/文本后精确比较
- **数值容差**：浮点数允许 ±0.15 误差
- **关键词重叠**：集合相似度 ≥70% 即判定正确

## 4. 实验结果

### 4.1 总体结果

| 模型 | A（直答） | C（oracle） | B（真实） | A-C（额外步骤） | C-B（不对齐） | A-B（总损失） |
|------|---------|-----------|---------|--------------|-------------|-------------|
| **0.5B** | 29.4% | 23.5% | 17.6% | **+5.9%** | **+5.9%** | **+11.8%** |
| **1.5B** | 38.2% | 44.1% | 35.3% | **-5.9%** | **+8.8%** | **+2.9%** |
| **3B** | 32.4% | 61.8% | 26.5% | **-29.4%** | **+35.3%** | **+5.9%** |

> 注：A-C 为负值表示 oracle 委派**优于**直答，说明任务分解对模型有帮助；A-C 为正值表示委派反而降低性能。

### 4.2 结果解读

#### 发现 1：目标不对齐在所有模型上均存在

C > B 对所有三个模型成立，说明**预言机（完美子代理）始终优于真实子代理**，目标不对齐是跨尺度的结构性问题。

#### 发现 2：目标不对齐随模型规模增长而加剧（反直觉）

```
misalignment 随模型规模变化：
  0.5B:  +5.9%  (misalignment 最小)
  1.5B:  +8.8%  (+49% 增长)
  3B:   +35.3%  (+301% 增长 vs 0.5B)
```

预期结论：更强的模型应该能更好地理解委派意图、更好地综合子代理输出，从而 misalignment 更小。但实验结果相反——**3B 的 misalignment 损失是 0.5B 的 6 倍**。

可能的解释：
- 3B 模型生成了更多中间推理步骤，引入更多格式/语义变异
- 3B 模型的子代理输出更丰富（temperature=0.3），增加了父代理解析的不确定性
- 0.5B 的 oracle/真实子代理输出高度相似（两者均极简短），因此对齐损失小

#### 发现 3：任务分解对 0.5B 有害无利

```
0.5B 的 A-C = +5.9%（正值）：
  → 即便子代理输出完美，委派步骤也降低了准确率
  → 分解任务对 0.5B 始终有害
  → 建议：0.5B 模型禁用强制 subagent 审查/委派
```

#### 发现 4：1.5B 是唯一从委派中净获益的模型

```
1.5B 的分解净收益分析：
  - 额外步骤收益（A-C）：-5.9%（委派步骤带来 +5.9pp 收益）
  - 不对齐损失（C-B）：+8.8%
  - 净收益（A-B）：+2.9%（勉强正收益）
  
1.5B 的 oracle 委派（C）大幅优于直答（A），说明：
  → 任务分解对 1.5B 有真实的推理帮助
  → 但不对齐损失抵消了大部分收益
  → 如果能消除不对齐，1.5B 的委派收益可达 11.8pp（44.1% - 32.4%）
```

#### 发现 5：3B 的 oracle 委派带来巨大收益，但被对齐损失完全抵消

```
3B 的分解效果分析：
  - oracle 委派（C）= 61.8%，比直答（A=32.4%）高出 29.4pp
  - 这意味着：3B 有强大的多步推理能力，分解任务后大幅受益
  - 但真实委派（B）= 26.5%，比直答还低 5.9pp
  - misalignment（C-B）= 35.3pp：几乎完全抵消了 oracle 带来的收益

这揭示了一个关键问题：
  → 3B 有能力从任务分解中大幅受益（+29.4pp）
  → 但实际上无法获益，因为 misalignment 损失更大
  → 如果能解决 misalignment，3B 的委派表现可达 61.8%（几乎翻倍）
```

## 5. 按任务类型分析

### 5.1 各类型不对齐强度

以下为 3B 模型上各任务类型的典型不对齐案例：

| 类型 | 典型不对齐案例 | 机制 |
|------|-------------|------|
| **Sequential Logic** | sl_02：三段条件链，subagent 复述给定条件，父代理无法完成链条 | 链长度增加对齐难度 |
| **Comparative Analysis** | ca_03：百分比问题，subagent 给出 99%，父代理忘记比较第二个场景 | 部分答案导致父代理过早收敛 |
| **Two-Part Questions** | tp_07：密度×体积=质量，子代理给出体积，父代理在单位换算上出错 | 链式操作中信息丢失 |
| **Constraint Satisfaction** | cs_01：子代理列出候选集，父代理在多约束过滤时丢失部分条件 | 额外过滤步骤引入噪声 |

### 5.2 不对齐的结构性来源

从 3B 的案例分析中，我们识别出三类主要不对齐来源：

**信息丢失（Information Loss）**：
- 子代理输出格式与父代理预期不符（如"120 km" vs "120"）
- 单位在传递过程中被丢失或误解析
- 多位数数字在传输中丢失精度

**语义漂移（Semantic Drift）**：
- 子代理对子问题的回答与父代理所需语义存在偏差
- 如：子代理给出"步进电机"的转速，但父代理需要的是"转速单位"

**推理链断裂（Reasoning Chain Break）**：
- 子代理输出是父代理下一步推理的前提，但父代理无法将两者串联
- 如：子代理给出"15"，父代理需要的是"15 是某个数的平方根"

## 6. 对 tiny-agents 框架的启示

### 6.1 框架设计建议

基于实验结果，提出以下框架设计建议：

```
✓ 建议：对 1.5B 和 3B 模型启用任务分解委派
    → 两者均有 oracle 委派净正收益（1.5B: +2.9%, 3B 潜在 +35.3%）
    → 但 3B 需要对齐机制才能实现收益

✗ 建议：对 0.5B 模型禁用强制 subagent 委派
    → 0.5B 的委派对所有任务类型均有害
    → 额外步骤损失 + 对齐损失 > 委派收益

⚠ 建议：对 3B 模型必须实现对齐机制
    → 否则委派反而降低性能（-5.9pp）
    → 可选方案：结构化输出约束、critic 审查、父子代理显式对齐
```

### 6.2 可行的对齐改进方向

```
1. 结构化输出约束（Format Enforcement）
   子代理输出格式标准化（如 JSON Schema、纯整数不带单位）
   预期效果：可消除 format_mismatch 类不对齐

2. Critic 审查机制（Critic Review）
   父代理生成答案后，由独立的 critic 代理审查并纠正
   预期效果：降低语义漂移，但引入额外步骤成本

3. 父子代理显式对齐（Explicit Alignment Protocol）
   父代理在委派前向子代理传达完整的任务上下文和期望格式
   预期效果：减少信息丢失，但增加 prompt 长度

4. 置信度感知委派（Confidence-aware Delegation）
   父代理先评估自己回答的置信度，低置信度时才委派
   预期效果：避免 irrelevant_delegation 类型的资源浪费
```

### 6.3 任务类型路由建议

```
建议路由策略：

if task_type == "Single-step Knowledge Retrieval":
    → 使用 0.5B 直答（委派无收益）
elif task_type == "Two-Part Knowledge + Reasoning":
    → 使用 1.5B 或 3B + 对齐机制
elif task_type == "Multi-step Math":
    → 使用 3B + oracle 子代理（如可行）
elif task_type == "Constraint Satisfaction":
    → 使用 3B + 结构化输出约束
else:
    → 评估 misalignment 风险后决定
```

## 7. 局限性与未来工作

### 7.1 局限性

1. **模型规模**：仅测试了 0.5B / 1.5B / 3B 三个规模，未覆盖更大模型（7B+）作为对比
2. **任务数量**：34 个任务在统计上有意义，但可能无法完全覆盖真实世界的任务多样性
3. **评估方法**：基于规则的评估（正则匹配+容差）可能对某些边界案例产生误判
4. **Temperature 设置**：使用 temperature=0（deterministic），随机采样场景下的结论可能不同
5. **同构代理**：父子代理使用相同模型（homogeneous agents），异构代理场景（不同模型规模）未测试
6. **单步委派**：仅测试了单步委派场景，多层嵌套委派的行为未探索

### 7.2 未来工作

```
1. 异构代理实验
   父代理 3B + 子代理 0.5B 的组合是否能work？更强的父代理能否弥补子代理弱点？

2. 对齐机制消融实验
   在 3B 上分别测试：结构化输出 vs Critic 审查 vs 显式对齐协议
   验证哪种机制最能降低 misalignment

3. 嵌套委派场景
   父代理 → 子代理 → 孙代理，多层委派的对齐损失如何累积？

4. 跨任务类型泛化
   在一种类型上训练的对齐策略，能否泛化到其他类型？

5. 更大模型对比
   将 3B 结果与 7B/14B 进行对比，验证"更大模型 misalignment 更小"的假设
```

## 9. v2 消融实验：带上下文的委派（Mode D）

### 9.1 实验动机

v1/v4 实验揭示了 C-B gap（oracle vs 真实委派）的存在，但无法回答：**真实委派失败，究竟是因为父代理问了错误的问题，还是因为子代理答错了问题？**

为了分离这两个因素，我们设计了 Mode D：给子代理提供完整的**原始问题上下文**（original question），让子代理能够理解父代理真正需要什么。

### 9.2 Mode D 实验设计

```
Mode A（直答基线）：问题 → 模型直答 → 准确率 A%
Mode B（真实委派）：问题 → 父代理问子代理 → 真实子代理回答 → 父代理综合 → 准确率 B%
Mode C（oracle委派）：问题 → 父代理问子代理 → 注入oracle答案 → 父代理综合 → 准确率 C%
Mode D（带上下文的委派）：问题 → 父代理问子代理 + 提供原始问题上下文 → 真实子代理回答 → 父代理综合 → 准确率 D%
```

**关键设计**：在 Mode D 中，我们将原始问题（original question）作为额外上下文注入子代理的 prompt，使子代理能够理解父代理正在尝试解决什么任务。

**预期**：如果 D > B，说明子代理失败的原因是**缺少上下文**；如果 D ≈ B，说明子代理失败的主要原因是父代理问了错误的问题。

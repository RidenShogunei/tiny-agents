# GoalMisAlignBench：第一阶段实验报告

> **GoalMisAlignBench**：在多智能体系统中，将目标错位与额外步骤损失进行分离  
> 模型：Qwen2.5-3B-Instruct | 数据集：60 个案例 | 日期：2026-04-24

---

## 1. 执行摘要

当父代理将子任务委派给子代理时，准确率下降主要有两个不同来源：

1. **额外步骤损失（Extra-step loss）**：即便子代理输出完美，增加一个推理步骤也会引入噪声。  
2. **目标错位损失（Goal misalignment loss）**：子代理在局部上正确，但与父代理真正需要的信息不一致。

**关键结果**：委派总损失为 23pp。**其中 78% 的损失（18pp）来自额外步骤成本，而非目标错位（5pp）。** 但这一比例在不同错位类型间差异显著，说明目标错位是一个**集中出现的结构性问题**，而非多智能体系统的均匀属性。

---

## 2. 方法论

### 三模式实验设计

三个模式都使用同一个 Qwen2.5-3B 模型，并在同一组 60 个案例上评估：

```
┌─────────────────────────────────────────────────────────────────────┐
│  模式 A（基线）                                                      │
│  问题："Python 1991, Java 1995, which released first?"             │
│  → 模型直接回答："Python"                                            │
│  准确率：52%（31/60）                                                 │
└─────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────┐
│  模式 B（真实委派）                                                  │
│  父代理 → 子代理："In what year was Java released?"                │
│  子代理回答："1995"                                                   │
│  父代理用 1995 回答原问题："Java"  ← 错误                            │
│  准确率：28%（17/60）                                                 │
└─────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────┐
│  模式 C（预言机委派）                                                │
│  父代理 → 子代理："In what year was Java released?"                │
│  我们给父代理真实答案："1995"  ← 预言机                              │
│  父代理回答："Java"  ← 仍然错误（与真实委派一致）                    │
│  准确率：33%（20/60）                                                 │
└─────────────────────────────────────────────────────────────────────┘
```

### 为什么模式 C 很重要

如果 C = A，说明委派本身无害，全部损失来自子代理质量。  
如果 C = B，说明全部损失来自子代理质量（错位 ≈ 0）。  
如果 C > B，说明存在错位（预言机优于真实子代理）。  
如果 C < A，说明即便子代理完美，委派仍会带来伤害。

### 损失分解

```
extra_step_loss  = A% − C%    ← 委派步骤本身带来的伤害
misalign_loss    = C% − B%    ← 纯目标错位损失
total_loss       = A% − B%
```

## 3. 总体结果

```
模式 A（单 3B，无子代理）：                 52%  [基线]
模式 C（3B + 预言机子代理输出）：            33%  ← 额外步骤成本：+19pp
模式 B（3B + 真实子代理）：                  28%  ← 错位额外增加：+5pp
────────────────────────────────────────────────────────────────
总委派损失：                                +24pp
  其中：额外步骤成本                         +19pp  (79%)
  其中：纯目标错位                            +5pp   (21%)
```

**结论**：对于 Qwen2.5-3B，主导性成本来自额外推理步骤，而不是代理间目标错位。

## 5. 案例分析

### 5.1 `partial_delegation` — gm51：Wednesday vs Thursday

**问题**：`"Wednesday"`（9 个字母）和 `"Thursday"`（8 个字母），哪个更长？

**子代理委派**：`"Wednesday"` 有多少个字母？

```
预言机子代理输出：  "9"
真实子代理输出：    "5"  ← 完全错误（可能把 "Wed" 计成 3？）

模式 A（直答）：    "Wednesday"  ✓  （模型可直接读到题干里的 9）
模式 C（oracle=9）："Wednesday"  ✓  （9 > 8，正确）
模式 B（real=5）：  "8"          ✗  （模型认为 Thursday=8 > Wednesday=5）
```

**机制**：真实子代理字母计数失败。父代理收到 `"5"` 后，与题干中可读到的 Thursday=8 比较，错误选择了 Thursday。这里是**子代理质量问题直接导致父代理出错**；但由于 oracle=9 时父代理仍可答对，因此 C 与 B 出现分歧，这正是 C−B > 0 所刻画的现象。

**为何属于错位**：子代理输出（5）在局部看似“合理”（例如把 Wednesday 理解成 5 天工作周），但对当前任务语义上是错误的。

---

### 5.2 `format_mismatch` — gm01：珠穆朗玛峰 vs K2

**问题**：珠穆朗玛峰 8848 米，K2 8611 米。哪个更高？

**子代理委派**：K2 的海拔是多少米？

```
预言机输出：  "8611"   （干净、精确的数字）
真实输出：    "8,848"  ← 子代理把珠峰高度当成了 K2！

模式 A（直答）：       "Mount Everest"  ✓  （模型读到 8848 > 8611）
模式 C（oracle=8611）："Mount Everest"  ✓  （比较正确）
模式 B（real=8,848）： "K2"             ✗  （模型看到 8848 > 8611，却选了 K2）
```

**机制**：真实子代理混淆了两座山对应的高度（珠峰 8848、K2 8611），把 `"8,848"` 作为 K2 返回。父代理收到 `"8,848"` 后，与题干中的 Everest=8848 比较，得出 K2 更高这一错误结论——属于**伪精确（false precision）错误**。

**关键发现**：即便格式是干净的（oracle），模型表现也**优于**直接作答（90% > 80%）。这表明，以标准格式提供精确数字（无逗号）有助于模型正确比较；而题干中的原始数字上下文可能被误读。

**工程启示**：对于这类问题，约束子代理输出格式（如“输出纯整数，不带逗号、不带单位”）很可能能抹平全部错位差距。

---

### 5.3 `context_collapse` — gm35：现在几点？

**问题**：钟表上，时针正指向 3，分针正指向 12。现在是几点？

**子代理委派**：当分针指向 12 时，时针在哪？

```
预言机输出：  "3"  （小时数字）
真实输出：    "3"  ← 此处子代理恰好答对

模式 A（直答）：      "3:00"  ✓  （正确）
模式 C（oracle=3）：  "3:00"  ✓  （oracle 给 "3"，模型推断为 "3 点整"）
模式 B（real=3）：    "3"     ✗  （模型只输出 "3"，而非 "3:00"）
```

**机制**：子代理对“时针在 3”回答正确；但父模型拿到 `"3"` 后，把它直接当最终答案输出为 `"3"`，而不是 `"3:00"`。这就是**上下文坍塌（context collapse）**：子代理输出对子问题正确，但缺少作为最终答案所需的时间语境。

**这是最纯粹的错位案例**：即便子代理技术上正确（"3"），父代理仍会给出错误最终答案。

---

### 5.4 `semantic_drift` — gm11：Strawberry

**问题**：`"Strawberry"` 中，`'r'` 和 `'b'` 哪个出现更多？

**子代理委派**：`"Strawberry"` 中 `'r'` 出现几次？

```
预言机输出：  "3"
真实输出：    "2"  ← 子代理计数错误

模式 A（直答）： "r"   ✓  （模型直接数得 r=3, b=2）
模式 C（oracle）："3"  ✗  （模型输出 "3"，而不是 "r"）
模式 B（real）：  "2"  ✗  （模型输出子代理的错误计数）
```

**此案例呈现 A✓ 但 C✗**：与直接作答相比，预言机步骤反而**有害**。原问题问的是“r 还是 b”，但委派子问题问的是“次数”。预言机给出正确次数（3）后，父代理却输出了 `"3"` 而非 `"r"`。这说明：**委派错误的子问题，有时比不委派更糟。**

---

### 5.5 `precision_loss` — gm27：苹果 vs 橙子定价

**问题**：一个苹果价格在 $1.00 到 $1.50 之间，橙子每个 $1.20。哪个更贵？

**子代理委派**：一个苹果可能的最高价格是多少？

```
预言机输出：  "1.50"
真实输出：    可能是 "about $1.25" 或 "$1.00-$1.50" 等区间表达

模式 A（直答）：  "Orange"              ✓  （模型比较 1.20 与 [1.00,1.50]）
模式 C（oracle）： "Orange costs more." ✓  （1.50 > 1.20）
模式 B（real）：   "Apple"              ✗  （真实子代理给区间，父代理混淆）
```

**机制**：题目给的是“between $1.00 and $1.50”。子代理回答 `"1.50"`（最大值）在技术上正确，但丢失了“区间”语义。父代理比较 1.50 和 1.20，从而得出橙子更贵。真实子代理则可能返回区间；父代理不知应取最小值/最大值/中点，因此选成苹果。

---

### 5.6 `irrelevant_delegation` — gm41：Python vs Java（“看似无害”）

**问题**：Python 1991、Java 1995。哪个发布更早？

```
模式 A："Python"  ✓  （题干给出两年，直接比较即可）
模式 C："Python"  ✓  （oracle=1995，父代理比较 1991 vs 1995 → Python）
模式 B："Python"  ✓  （真实子代理=1995，逻辑同上）
```

该案例三种模式都答对了，但在 B/C 中是**“答对了但理由错了”**。父代理本应直接根据题干作答（1991 < 1995），但在 B/C 中却依赖了子代理输出。因此这里的委派是“偶然有效”。

## 7. 关键发现

### 发现 1：额外步骤成本主导总损失

总委派损失中的 79% 来自额外推理步骤本身，而不是子代理质量或目标错位。这一点有些反直觉：人们可能认为“目标错位”（两个代理目标不同）才是主失败模式，但对小模型而言，主问题是**只要多加一步，性能就会下降**。

### 发现 2：`irrelevant_delegation` 是最浪费的类型

当问题可直接回答时，强行模拟委派步骤会使准确率下降 40pp。直接工程含义是：**能直接解的问题，不要上多智能体。**

### 发现 3：`context_collapse` 会导致灾难性失败

`context_collapse` 与 `partial_delegation` 都呈现 >30pp 的步骤成本，意味着即便子代理完美，当原问题需要多步或比较推理时，父模型仍无法正确使用子代理输出。

### 发现 4：`format_mismatch` 是唯一“委派有益”的类型

当子代理输出格式可标准化（纯整数、无标点）时，预言机可带来 10pp 提升（C=90% > A=80%）。这提供了**接口设计可逆转错位损失**的最强证据：若能约束子代理返回形式，委派反而能变成收益项。

### 发现 5：目标错位是“集中出现”的

仅有 6 个案例（10%）呈现 C✓ B✗，即真实子代理输出导致失败，而预言机本可成功。这说明目标错位**确实存在，但集中在特定结构模式**：格式错配（gm01）、时间语义混淆（gm35）、部分信息缺失（gm51）。

## 8. 系统设计启示

### 建议做（Do）

- 将委派用于**外部知识获取**（网页搜索、数据库检索），即父代理缺少上下文的场景  
- 通过提示工程**约束子代理输出格式**（JSON Schema、单位规范）  
- 对 `format_mismatch` 类问题，增加“格式干净”的委派步骤可**提升**准确率  
- 在选择单智能体/多智能体前，先测量真实任务的**类型分布**

### 不建议做（Don't）

- 对父代理已具备完整上下文的任务继续委派（irrelevant_delegation）  
- 在**多步链式推理**任务上直接使用多智能体（context_collapse 接近零准确）  
- 认为只要提升子代理质量就能弥合委派差距（步骤成本更大）  
- 未分析任务中错位类型分布就直接部署多智能体

## 9. 局限性与下一步

**局限性：**
- 仅测试了单一模型规模（3B）。更小模型（0.5B、1.5B）可能呈现不同步骤成本比例。  
- Temperature=0（确定性）。随机采样可能改变判定分布。  
- 评估器使用模糊匹配（Jaccard ≥ 0.6），可能误判部分边界案例。  
- 60 个案例在统计上有意义，但对真实世界任务多样性仍偏窄。

**计划实验：**
1. 模型规模扩展：0.5B / 1.5B / 3B 对比（更小模型错位是否更严重？）  
2. 跨类型泛化：在一种类型上训练，能否降低另一类型错位？  
3. 提示工程：结构化子代理输出（JSON）能否弥合 format_mismatch 差距？  
4. 能力差距：3B 父代理 + 0.5B 子代理（更弱子代理是否放大错位？）
